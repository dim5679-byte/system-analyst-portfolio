# Use Cases

## Назначение документа

Документ описывает основные сценарии использования системы Smart Home Management Platform.

Use Cases используются для детализации пользовательских действий, проверки требований и последующего проектирования API, BPMN и UML-диаграмм.

---

# UC-001 Регистрация пользователя

**Цель:** создать учетную запись пользователя.

**Основной актор:** новый пользователь.

**Связанные требования:** FR-001, FR-002, FR-003  
**Связанная User Story:** US-001

## Предусловия

- пользователь не авторизован;
- пользователь открыл экран регистрации.

## Основной сценарий

1. Пользователь открывает экран регистрации.
2. Пользователь вводит email и пароль.
3. Система проверяет корректность введенных данных.
4. Система создает учетную запись.
5. Система перенаправляет пользователя на экран входа.

## Альтернативные сценарии

### Некорректный email

1. Пользователь вводит email в неверном формате.
2. Система отображает сообщение об ошибке.
3. Пользователь исправляет email.

### Пользователь уже существует

1. Пользователь вводит email, который уже зарегистрирован.
2. Система отображает сообщение о существующей учетной записи.

## Постусловия

- учетная запись создана;
- пользователь может войти в систему.

# UC-002 Создание дома

**Цель:** добавить новый дом в систему.

**Основной актор:** авторизованный пользователь.

**Связанные требования:** FR-005  
**Связанная User Story:** US-002

## Предусловия

- пользователь авторизован.

## Основной сценарий

1. Пользователь открывает раздел домов.
2. Пользователь выбирает действие создания дома.
3. Пользователь вводит название дома.
4. Система сохраняет новый дом.
5. Новый дом отображается в списке домов.

## Альтернативные сценарии

### Не указано название дома

1. Пользователь оставляет поле названия пустым.
2. Система отображает сообщение об обязательном поле.

## Постусловия

- дом создан;
- пользователь может добавлять комнаты и устройства.

# UC-003 Создание комнаты

**Цель:** создать комнату внутри выбранного дома.

**Основной актор:** авторизованный пользователь.

**Связанные требования:** FR-008, FR-009  
**Связанная User Story:** US-003

## Предусловия

- пользователь авторизован;
- в системе создан хотя бы один дом.

## Основной сценарий

1. Пользователь выбирает дом.
2. Пользователь открывает раздел комнат.
3. Пользователь выбирает действие создания комнаты.
4. Пользователь вводит название комнаты.
5. Система сохраняет комнату.
6. Комната отображается в выбранном доме.

## Альтернативные сценарии

### Комната с таким названием уже существует

1. Пользователь вводит название существующей комнаты.
2. Система отображает предупреждение.
3. Пользователь изменяет название комнаты.

## Постусловия

- комната создана;
- пользователь может добавить устройство в комнату.

# UC-004 Подключение устройства

**Цель:** подключить устройство к системе.

**Основной актор:** авторизованный пользователь.

**Связанные требования:** FR-011, FR-013  
**Связанная User Story:** US-004

## Предусловия

- пользователь авторизован;
- в системе создан дом;
- устройство готово к подключению.

## Основной сценарий

1. Пользователь открывает раздел устройств.
2. Пользователь выбирает действие добавления устройства.
3. Пользователь сканирует QR-код устройства.
4. Система проверяет данные устройства.
5. Система добавляет устройство в список.
6. Пользователь выбирает комнату для устройства.

## Альтернативные сценарии

### QR-код не распознан

1. Система не может распознать QR-код.
2. Система отображает сообщение об ошибке.
3. Пользователь повторяет сканирование.

### Устройство уже подключено

1. Система определяет, что устройство уже добавлено.
2. Система отображает сообщение о невозможности повторного подключения.

## Постусловия

- устройство добавлено;
- пользователь может управлять устройством.

# UC-005 Управление устройством

**Цель:** изменить состояние подключенного устройства.

**Основной актор:** авторизованный пользователь.

**Связанные требования:** FR-014, FR-015, FR-016  
**Связанная User Story:** US-005

## Предусловия

- пользователь авторизован;
- устройство подключено;
- устройство доступно в системе.

## Основной сценарий

1. Пользователь открывает список устройств.
2. Пользователь выбирает устройство.
3. Пользователь изменяет состояние устройства.
4. Система отправляет команду устройству.
5. Система обновляет состояние устройства в интерфейсе.

## Альтернативные сценарии

### Устройство недоступно

1. Система не может отправить команду устройству.
2. Система отображает сообщение о недоступности устройства.

## Постусловия

- состояние устройства изменено;
- новое состояние отображается пользователю.
